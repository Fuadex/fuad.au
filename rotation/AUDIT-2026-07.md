# Rotation — Grounding Audit & Forward Plan (2026-07-04)

> Written after the July layout pass (Overview pulse row, 3:3:2 map band, Explore 6-col genre
> grid, artist-page recomposition). Companion to **ARCHITECTURE.md** (inventory) and
> **ROADMAP.md** (build queue + evaluated API catalogue). This file is the *assessment*:
> how good the platform actually is, where it's weak, and where the leverage is if the goal
> is "a digital library on steroids". Nothing here re-litigates ROADMAP's rejected sources.

---

## 1 · What this thing is, honestly assessed

**Strengths (real, worth protecting):**

- **The derived-insight philosophy is the moat.** Nothing else renders a personal library as
  obsessions, flameouts, incubations, taste geography, band mortality, or a record shop of
  spines. The brief in ARCHITECTURE §1 ("insight, not mirrors") has been enforced and it shows.
- **The data pipeline is genuinely strong.** One CSV of truth (319k rows), 20+ enrichment
  caches that degrade gracefully, an 11-second build, CI deploys with a smoke gate, daily
  scrobble sync, weekly enrichment. Most hobby projects never get here.
- **The lazy-tier design works.** First paint pays for music-data.js + eager JSX; the other
  ~19 MB (media-index, track-audio, flows, geo, calendar detail…) loads on demand. The
  boundary-per-view crash isolation has kept single-page bugs from blanking the site.
- **Identity plumbing** (slug ids, alias resolution, canonical subgenres, family hues) is
  consistent across build and runtime — the reason cross-view navigation "just works".
- **Buildless is a feature for this project.** One person, no toolchain drift, `git push` =
  deploy, any file is view-source-debuggable in production.

**Structural weaknesses (ranked by real cost):**

1. **First-paint payload.** Tier 0 = music-data.js (~4.4 MB) + ~500 KB of JSX compiled
   *in the browser on every load* by @babel/standalone (itself ~2.5 MB) + React UMD. On a
   cold mobile connection this is seconds of parse/compile before anything renders. It's a
   deliberate choice (ROADMAP A1), but it is now the single biggest UX cost, and it grows
   with every feature. Mitigations that keep the buildless *authoring* model are in §3.
2. **`window`-global module system.** Components communicate via `Object.assign(window, …)`
   and load-order in index.html. It works, but every new file is an implicit dependency-graph
   edit, nothing is tree-shakeable, and name collisions fail silently.
3. **CSS is scattered and override-driven.** Inline styles + per-component `<style>` blocks +
   `!important` media overrides (m-stack, the ≥981/≥1150 blocks). The June/July breakage class
   — `.ov-calslot` collapsing to a 1/12 sliver because a wrapper wasn't an `.r-card` — is a
   direct product of this: layout truth lives in three places (inline, component CSS, global
   utilities) with different specificity. Breakpoints are also inconsistent per file
   (860 / 980 / 981 / 1100 / 1150 / 1250).
4. **`rotation-views2.jsx` is a 92 KB four-page monolith** (artist/album/track/live). Already
   flagged in ROADMAP A5; still true; every artist-page change risks the other three pages.
5. **No URL state for the map band** (mode/focus/year/genre) — Explore got this, the map
   didn't. Deep-linking a filtered map view is impossible, and browser Back does nothing there.
6. **unpkg remains a single point of failure** (SRI-pinned, so a compromise is caught, but an
   outage takes the whole site down). Self-hosting three files erases the risk.
7. **Testing = smoke.js.** It gates deploys (good) but asserts data-shape, not rendering.
   There's no layout regression net at all — which is why overview breakages keep being
   discovered by eye. The `.sptmp/overflow-audit.js` puppeteer harness exists but is manual.

## 2 · Architecture verdict

Keep the architecture. It is unusually well matched to its constraints (one author, static
hosting, testing-in-prod, design-playground ethos). The right moves are *within* the model:

- **Precompile as an option, not a conversion.** A `precompile.js` step in CI (babel-transform
  each .jsx → .js at deploy time, swap script tags in the staged `_site/` only) keeps local
  authoring 100% buildless while production loses the 2.5 MB babel download AND the per-load
  compile. This is the highest-leverage perf change available and it's ~1 session. Local dev
  still loads .jsx via babel exactly as today.
- **Split music-data.js by first-paint need.** Overview needs TOTALS/NOW/TREND/INSIGHTS-lite/
  ARTISTS-lite; it does not need 1,572 albums, full EXPLORE, or CLOCK_BY_YEAR at t=0. A
  `music-core.js` (~1 MB) + `music-rest.js` (lazy, fetched immediately after first paint)
  would halve time-to-interactive without deleting anything.
- **`defer` on data scripts** — free win; parse stops blocking paint.
- **Views2 split** (artist / media / live) next time the artist page is touched — the July
  recomposition made it bigger again.
- **A tokens block in rotation-core** for breakpoints (`--bp-stack: 860px` etc. can't be used
  in media queries, but a single documented constant table + one naming convention can) and a
  rule: *wrappers that live in a bento grid must carry an `.r-card`-equivalent class* so the
  global stacking rules reach them. That rule would have prevented the calslot bug outright.
- **Self-host react/react-dom/babel** (or drop babel entirely once precompile ships).
- **Map URL state** — serialize `mode/focus/sel/filt/year` into the hash like Explore does.

## 5 · API landscape check (July 2026)

Verified working / integrated sources and the rejected list live in ROADMAP §② — still
accurate. What changed or was newly checked this pass:

- **Spotify tightened again (Feb 2026).** Beyond the Nov-2024 removals (audio-features,
  recommendations — already replaced by the archive dump), the February 2026 changes removed
  more catalogue endpoints (new releases, artist top tracks, popularity/follower fields) and
  **dev-mode now requires a Premium account with limited test users**; library save/remove
  endpoints moved to a generic `/me/library` keyed by URIs. **Impact on M3:** the *extended
  streaming history export* path (a privacy-data request, not the API) is unaffected and is
  now clearly the primary plan; the "quick liked-songs via `/me/tracks`" fast tier needs
  re-verification against the migration guide before building anything on it.
- **ListenBrainz — upgrade from "interesting, unscheduled" to genuinely useful.** Free API
  with statistics endpoints, MBID metadata lookup (name→MBID mapping at scale), and a
  growing social/discovery layer (similar-users, **Fresh Releases**). Two concrete uses:
  ① mirror scrobbles there once (they import from last.fm) and gain percentile/context stats
  plus a second-source backup of the listening history; ② **Fresh Releases filtered by the
  library** replaces the Spotify new-releases endpoint that no longer exists — a "new records
  from artists you play" feed, which the site currently has no source for.
- **Odesli/song.link** — free universal-link API (keyless at 10 req/min, key by email).
  One call per album/track at *build time* (or lazily, cached) yields open-in-your-platform
  links (Spotify/Apple/YouTube/Tidal/Bandcamp) for Reader, album and track pages. Cheap,
  additive, no auth burden.
- **Wikimedia Pageviews API** — keyless, per-article daily views since 2015 via a single REST
  call. Joined through the Wikidata QIDs we already planned to pull (MB→Wikidata links), this
  gives a **cultural-momentum axis per artist**: "you vs the world" (your plays vs global
  attention), "you were early" (your first scrobble vs their pageview breakout), obscurity
  drift over time. Batchable for the top-400 in one enrichment run.
- **Deezer** — stays dropped; confirmed no new API tokens are being issued in 2026.
- Everything in the rejected table (AcousticBrainz, Bandsintown, Songkick, Spotify
  audio-features, generic cover dumps) re-checked by reputation only — no revival signals.

- ◆ **Search operators** in the `/` overlay: `year:2019`, `genre:dnb`, `country:JP`,

- ● **residences.json** (10 minutes of manual entry) — still the single cheapest unlock for

**Cross-referencing (the "on steroids" part)**
- ▲ **Covers & language layer** (approved): MB full dump × 39k ISRCs → cover-of relations +
  work language. Feeds "your favourite covered artist", the JP-arc story, a language
  timeline. Highest-value pending data job after pins.json.
- ▲ **Collab-graph shipping**: 5,730 credit edges already extracted, unshipped. Play-filter
  at build → "scene webs" (who actually connects your library) + a network-centrality story.
- ▲ **Wikipedia momentum join** (new, §5): plays × pageviews percentile per artist; "caught

- ▲ **ListenBrainz context**: your artist/track counts vs the LB population — real
  percentiles instead of last.fm's global listener counts alone.

**New material (recommendation surfaces)**
- ▲ **"New from your library" feed**: ListenBrainz Fresh Releases filtered to played/similar
  artists — fills the dead Spotify new-releases hole; a natural Overview module and the
  first *forward-looking* surface on a site that's currently all history.
- ◆ **Dig-list generator**: rank the 6.5k shrinkwrapped LPs by (artist affinity × archive
  album popularity × completeness gap × mood match to your recent 30 days) → a weekly
  "pull these five records" card. All inputs already exist in caches; this is pure build-data.
- ▲ **Odesli links** on Reader/album/track pages so every recommendation is one click from
  playable anywhere.

**Analytics & self-knowledge (M4, sharpened)**
- ◆ Binge signature, repeat-gap distribution, artist half-life curves — all CSV-derivable now.
- ▲ Weather × listening once residences.json exists (Open-Meteo, keyless backfill).
- ● Gigs module (M2) still blocked only on the setlist.fm account — it remains the highest
  *emotional*-value pending feature ("caught them in time").

**Platform**
- CI precompile + music-data split (§2/§3) before any of the above adds more weight.
- Wrapped mode + share-card PNG (approved, unstarted) — pairs naturally with the new
  analytics the moment they exist.

## II.0 · Measured payload (2026-07-07)

36 deployed files, **33.9 MB raw / 11.6 MB gzipped** (Pages serves gzip). The split:

| Tier | Raw | Gzip | Notes |
|---|---|---|---|
| Runtime (unpkg): react+react-dom+babel | ~3.2 MB | ~0.7 MB | every cold load |
| Tier-0 data: music-data.js + live-data.js | 4.8 MB | 1.44 MB | every load (daily cache-bust) |
| Tier-0 JSX (14 files, in-browser compile) | 580 KB | 162 KB | every load + compile CPU |
| Lazy tier (20 files) | 28.5 MB | 9.9 MB | on demand — healthy |

**Cold first paint ≈ 2.3 MB gzipped + the Babel compile.** Since Part I: music-data grew
3.5 → 4.8 MB (kept-artists 200→400, GIGS, LANGUAGE/MOOD/THEMES/LINEUPS/TOUR insights) and the
text layer added ~5.6 MB raw — **all of it lazy**, which is the weight discipline working.
Biggest lazy files: media-index 7.0 MB (2.5 gz) · calendar-detail 3.6 (1.0) · artist-flow 2.9
(0.55) · track-audio 2.7 (1.0) · track-previews 2.4 (1.34) · artist-detail 2.4 (1.0) ·
llm-about 2.0 (0.66).

**Optimization verdict + queue (supersedes Part I §3 priorities only where noted):**
1. **CI precompile** stays #1 (unchanged): kills the 700 KB babel download *and* the compile.
2. **music-data split** stays #2 (core ~1 MB for Overview; rest fetched post-paint).
3. **NEW — llm-about sharding.** 660 KB gz loads on the *first track page* to show one ~300-byte
   blurb, and grows with every batch. Split by artist-slug first letter (~30 shards of ~65 KB)
   or hot-1000/rest. Same pattern fits genius-themes-lazy (458 KB gz) if it keeps growing.
4. **NEW — retire blurb-demo.js** (20 KB but loaded on every TrackView *alongside* llm-about;
   it's the superseded bake-off set — fold its winners into llm-about, delete file + both loaders).
5. `defer` on the two data scripts; self-host runtime (also §II.5).
6. Watch media-index: 19.5k full cover URLs = ~2.2 MB of the file is `https://i.scdn.co/image/…`
   strings. A one-line prefix table (`S:` → `https://i.scdn.co/image/`) would cut ~500 KB gz
   across media-index + music-data with a 5-line decoder in GenCover.

## II.1 · Architecture & data convergence

**Verdict unchanged: keep it.** The modular lazy-tier + one-CSV-of-truth + graceful-cache
design is the reason the site absorbed a text layer, a gigs domain, and a tour explorer in one
week without breaking. Convergence findings:

- **Parallel per-track dictionaries** (track-audio, track-previews, llm-about, genius-mood/
  about/themes-lazy) all key on `artistSlug~trackSlug`. This is *duplication of keys, not of
  data* — each file serves a different view moment. Converging them into one blob would force
  TrackView to pay for Shelves' data and vice versa. **Keep the split; converge the loader**:
  one `useLazy("track-audio")` hook instead of six hand-rolled `document.createElement` blocks
  (they differ subtly — some have `onerror`, some don't, one double-loads blurb-demo).
- **Real duplication worth fixing:** cover URLs appear in both music-data (THUMBS/SPOTIMG,
  10.2k) and media-index (19.5k) — the overlap is most of THUMBS. The prefix-table fix (II.0.6)
  halves the cost without restructuring. Track yearTail/plays exist in both ROTATION.TRACKS
  (top 50) and media-index — trivial size, ignore.
- **The gigs-map choke — diagnosed** (TourMap, rotation-views3.jsx). Pan/zoom store the
  transform in React state, so **every pointermove/wheel frame re-renders the entire SVG**:
  ~150 land paths + all city dots + route polylines, each with `strokeWidth={1/rk}`-style
  per-child recomputes. Fix (one session): apply the transform imperatively to the `<g>` ref
  during the gesture and commit to state once on pointerup/wheel-settle; wrap the land layer
  in `React.memo`; use `vector-effect="non-scaling-stroke"` instead of `/rk` arithmetic where
  possible. Same pattern should then be audited in MapView (worldmap) which shares the idiom.
- Still open from Part I: views2 split (grew again — now 115 KB), map-band URL state, the
  bento-wrapper CSS rule, breakpoint consolidation (860/980/981/1100/1150/1250/1500 now).

## II.2 · UX

The July passes fixed the big compositional complaints. What's left, ranked by felt impact:

1. **First-load wait is the #1 UX item** — it's II.0.1/2, not a design change. On a cold
   mobile load the boot screen sits through ~2.3 MB + compile. After precompile+split it's
   sub-second on wifi.
2. **Perceived-progress**: the boot line ("reading 319k scrobbles…") is charming but static.
   Cheap win: tick it through phases (runtime → data → compile → paint) since script onloads
   are observable.
3. **Missing-data honesty** (Part I A4, still open): tracks/artists without audio rows
   silently lose radar/quadrant cards. A one-line "no audio data for this one" reads as data,
   not a bug.
4. **Back-button coverage**: Explore serializes to the hash; the Overview map band (mode/
   focus/year/genre) and Shelves (lens) don't. Both are one-session jobs and make deep links
   shareable.
5. **Discoverability of the text layer**: the What-it's-about switcher (Haiku/Sonnet/Opus/
   Web/Genius) is one of the site's richest features and is invisible until you open a track
   page. Surface it: a "read of the day" card in the Overview insight feed (provider exists
   pattern-wise), and/or a 📖 glyph on track rows that have reads.
6. **Percentile phrasing** — see II.4 (content).
7. Small frictions: search has no keyboard hint on mobile; calendar hour-clock can't filter
   (data-limited, §II.6); `prefers-reduced-motion` isn't respected by the spinning vinyl /
   pulse animations — one CSS block fixes it.

## II.3 · Functionality expansion (reviewed against ROADMAP §③ + Part I §6)

Everything in ROADMAP M1–M7 and Part I §6 stands — the queue is genuinely good. Net-NEW ideas
this pass (all "insight, not mirrors"-compliant), roughly ordered by value/cost:

- ◆ **Session & album-listen reconstruction.** Scrobble timestamps → sessions (gap-threshold);
  detect full-album front-to-back plays ("you've played *Lateralus* start-to-finish 14 times;
  your longest session was 9h41m on 2013-06-02"). Pure CSV derivation; feeds Stories + album
  pages + a heaviest-sessions ledger. The single richest untouched signal in the CSV.
- ◆ **Song-to-song segue graph.** What actually follows what (within-session adjacency,
  Markov-style): "after *Closer* you play *Reptile* 31% of the time". Top segues per artist on
  the artist page; a "your segues" Stories card. Same derivation pass as sessions.
- ◆ **Personal seasonality fingerprints.** Month-of-year profile per artist/genre ("Burial is
  a June artist — 74% of plays in AU winter"). One export; a "winter records" card, and a
  seasonal lens on Shelves.
- ◆ **Era auto-segmentation.** Change-point detection on monthly family/DNA vectors → your
  taste's actual chapters with boundary dates and "what changed" diffs — deeper than the
  hand-drawn ERAS. Ships as a Stories centrepiece.
- ◆ **Obsession lifecycle model.** You have OBSESSIONS/FLAMEOUTS/COMEBACKS as separate lists;
  fit the actual discovery→peak→decay curve per artist and *classify* trajectories (slow-burn,
  flameout, perennial, sleeper). Predictive twist: "current obsessions most likely to flame
  out, based on 20 years of your own curve shapes".
- ▲ **Data-health dashboard** (also serves the template, §II.7): a hidden `#health` view
  rendering enrich-coverage.js numbers — per-layer coverage, staleness, NONE-ledger counts.
  Turns cache drift from a session-time surprise into a glance.
- ◆ **Blurb coverage meter + reading mode**: "7,095 of your ~60k tracks have reads" with a
  shuffle-through-reads surface (the reads are good enough to browse for their own sake —
  Fuad already treats Web reads as first-class).

## II.4 · Content quality

- **"In your rotation, this track is…" — tighten.** Current gate is p≥72/p≤28 with "top N%"
  captions, so weak outliers ("danceable — top 26%") read as noise, and "slow — top 25%"
  is genuinely confusing (top of *what*?). Fixes, cheapest first: ① raise the gate to ≥85/≤15
  so only real outliers speak; ② reword to comparatives — "slower than 9 of 10 songs you
  play" (the percentile is already exactly this); ③ drop the module entirely when nothing
  clears the bar (partially true today); ④ stretch goal: anchor to a reference the reader
  knows — "about as slow as your average Boards of Canada".
- **Interpretations (llm-about, 7,095 tracks) — growth roadmap.** The tier system works
  (Haiku gist default → Sonnet/Opus for top-1000 → Web where the dump was blank). What it
  needs as it scales: ① a **promotion job** — when a track climbs into the top-1000, queue it
  for Sonnet+Opus (plays change, reads should deepen); ② a **spot-check pass** per batch
  (sample 5%, verify against lyrics — the JP web batches were verified this way, make it the
  norm); ③ a **style gate**: reject "this song explores themes of…" boilerplate at generation
  time; ④ keep the NONE ledger (.sptmp/blurb-nones.md) authoritative and retry classes when a
  new source appears (a JP lyrics DB unlocks 8 of the 12 uncovered).
- **Redundancy sweep (concrete):** ① blurb-demo.js — superseded, fold + delete (II.0.4);
  ② genius-about excerpts vs Web reads now overlap in role — keep both (human vs researched
  voices are the point) but the *Genius-first* default on bake-off tracks is stale;
  ③ "Where to dig" + "Your portrait" prose modules predate the text layer and read flat next
  to it — already greenlit for rework (A4), fold into the era-segmentation card when it lands;
  ④ the Sounds/Reads CJK caveat (n-gram NRC) still quietly mislabels some JP tracks — the
  planned transformer re-score (1f) fixes it, keep it queued.
- **Attribution check:** last.fm bios render without the "read more on last.fm" link that
  their ToS asks for — the artist page links to last.fm anyway, but move/verify the credit
  line sits with the bio text.

## II.5 · Resilience (external-dependency matrix)

Runtime dependencies, by blast radius:

| Dependency | Used for | If it dies | Severity | Mitigation |
|---|---|---|---|---|
| **unpkg** (react ×2, babel) | the entire runtime | **site fully down** | 🔴 critical | self-host the 3 files (they're static; React 18.3.1 UMD is frozen — React 19 dropped UMD, so this pin never updates anyway) |
| Google Fonts | typography | system-font fallback | 🟡 cosmetic | `font-display: swap` already implicit; optionally vendor woff2s |
| i.scdn.co (29.7k URLs) | album covers | GenCover generative fallback renders | 🟡 cosmetic | fallback already built; a quarterly sampled HEAD-probe would detect rot early |
| i.discogs.com / CAA / mzstatic | covers, artist imgs | same GenCover fallback | 🟡 cosmetic | same |
| p.scdn.co | 30-s previews | preview button dead | 🟠 feature loss | iTunes fallback already wired (Shelves + TrackView); previews are a deprecated Spotify asset class — **expect this one** |
| itunes.apple.com | preview fallback | needle-drop fallback dead | 🟡 | double failure needed |
| last.fm API (build-time) | daily scrobble sync | site serves stale-but-complete data forever | 🟠 → 🔴 long-term | the CSV is the archive (committed); **mirror to ListenBrainz now** — one import, free, second copy + future percentile stats |
| GitHub Actions/Pages | deploys | site static-frozen | 🟡 | any static host can serve `_site/` |
| Ticketmaster / setlist.fm | tour + gigs refresh | data ages gracefully | 🟢 | committed caches |

**Library/browser rot:** buildless is *good* here — no toolchain to rot. React 18 UMD +
Babel-standalone 7.29 are SRI-pinned, self-contained ES5/ES2018-class code; browsers do not
break that class of JS. The realistic browser-update risks are behavioral (the passive-wheel
event change already bit once — fixed natively in TourMap) and CSS/SVG rendering drift —
which the screenshot/overflow-audit tooling catches. **The single real structural risk is
unpkg availability**, and self-hosting erases it in an hour. Recommendation order:
① self-host runtime ② ListenBrainz mirror ③ vendor fonts ④ quarterly cover-rot probe.

## II.6 · Dynamism census

**Filter-reactive today:** the whole Explore surface (+URL state) · map Results + hours/
distinct-artists strip · calendar-year → map scrub · calendar day/week → Results ·
Gigs tour cross-filters (genre × map × calendar). **Static (build-time, all-time):** Overview
avg/day, %-of-history, heaviest day, streak · all TOTALS · most Stories cards · portrait ·
where-to-dig.

The unlock is one export: **`day-series.js` — per-day play counts** (~7.3k days × int ≈
25–40 KB gz, could even ride inside music-data). With it, avg/day, heaviest-day, streaks,
%-of-history all become computable client-side **under any date filter** — the exact ask
("avg/day of the filtered slice"). Second tier: a **day × family matrix** (7.3k × 12 small
ints ≈ 100–150 KB lazy) extends the same stats to genre filters. Third tier (already in
ROADMAP §5): per-day geography + per-day hour exports unlock calendar↔map-dots and
clock↔heatmap. Full per-day×artist is the only one that's genuinely too heavy — everything
else fits the budget. Stories staying mostly static is correct — they're narrative, not
dashboards; the Year-in-Review scrubber is the right amount of interactivity there.

## II.7 · Open-source template & copyright/risk

**Feasibility: high — the architecture is already 90% template-shaped.** CI builds everything
from a CSV + optional caches with graceful degradation; a fork with only a last.fm key would
produce a working site (generative covers, no lyrics layer) on day one. What M6 needs:
1. **config.json** — username, TZ, markets/cities, display name. The genre-FAMILIES hue map is
   personal taste taxonomy; ship it as an overridable default.
2. **Capability levels documented honestly:** L0 = last.fm key only (scrobbles, tags, bios,
   full core site) → L1 = free keyless enrichment (MusicBrainz, Wikidata, CAA, iTunes) →
   L2 = keyed APIs (Discogs, Ticketmaster, setlist.fm) → L3 = local dumps (large local Spotify,
   local Genius — most forkers will skip; everything they feed is optional).
3. **Split personal from code**: fuadex.csv, gigs*, all caches, llm-about/genius-* stay out of
   the template; code MIT, data none-of-anyone's-business.
4. The blurb pipeline can ship as *scripts* (bring-your-own-LLM-key), not data.

**Copyright/ToS exposure, ranked (for both fuad.au and any public template):**
- 🟠 **genius-about.json excerpts** — verbatim Genius editorial text, the one place the site
  republishes someone else's *prose*. On fuad.au: small excerpts, attributed + linked, personal
  scale — low practical risk. In a template/open-source repo: **exclude** (it's personal-cache
  anyway per #3). Long-term: Genius blocking WebFetch shows hardening; the public endpoint the
  chain uses could close — the full texts are already cached locally, so existing blurbs survive.
- 🟢 **Lyrics** — the site's strongest position: raw lyric text is **never shipped** — only
  derived stats (language, sentiment, themes) and original AI interpretations. Keep that rule
  absolute; it's what makes the whole text layer publishable.
- 🟡 **Hotlinked art/previews** (Spotify/Discogs images, p.scdn.co) — ToS-grey, universal
  practice; realistic risk is link-rot/hardening, not legal, and GenCover + iTunes fallbacks
  mean it degrades to cosmetic. Template inherits the same posture.
- 🟡 **last.fm bios** — need visible attribution (II.4 last point). Scrobbles themselves are
  your data (their API ToS explicitly supports personal use).
- 🟢 **AI reads** — original prose, lowest risk.
- **Likelihood board (24-month horizon):** previews vanish — likely; unpkg degrades — plausible
  (self-host preempts); Spotify image-hotlink hardening — possible (cosmetic); Genius About
  endpoint closes — possible (cached); last.fm API sunset — unlikely but existential for *new*
  data (ListenBrainz mirror is the hedge, and scrobbling itself can dual-target both).
