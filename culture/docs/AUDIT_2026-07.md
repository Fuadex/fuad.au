# Culture — site audit & forward roadmap (July 2026)

Grounding analysis of the current site: architecture, performance, weaknesses,
strengths — then a concrete plan for where to take it ("digital library on
steroids"). Numbers below were measured against the working tree at v=87.

## 1. Snapshot

| | |
|---|---|
| Seen library | **3,277 items** (206 curated in `data.js` + 3,071 imports) |
| Wishlist | **1,340 items** |
| By medium (seen) | Movies 1,163 · Shorts 1,007 · Games 359 · Feature Anim. 263 · Anim. Series 187 · TV 184 · Books 114 |
| App code | `culture-v2.jsx` 2,597 lines · `culture.css` 1,459 lines · no build step |
| Data payload | **13.0 MB raw / 3.65 MB gzipped** across 12 overlay+data `.js` files |

## 2. Architecture assessment

### What's genuinely good (keep)

- **Overlay-by-id model.** Hand-authored canon (`data.js` / `imports.js` /
  `wishlist.js`) + machine-owned overlays merged at runtime is a clean, robust
  separation. It survives regeneration, diffs beautifully in git, and makes every
  enrichment source independently rebuildable. This is the right shape for a
  personal dataset and should NOT be replaced with a database.
- **Git as the database.** Full history of every rating/note/badge decision, free
  hosting, trivial rollback, "push to master is fine." For a single-author site
  this beats any backend.
- **Resumable, cached fetchers.** Every `update_*.py` skips cache hits and respects
  quotas; `omdb_cache.json` deliberately committed so quota progress survives
  machine moves. `build_all.py` killing the forgot-to-bump bug was the single best
  workflow fix in the repo.
- **Perf care where it hurts:** spine-row virtualization, imperative scroll/thumb
  bookkeeping (zero React renders per frame), `loading="lazy"` images, production
  React builds. The interaction layer is in good shape.
- **Documentation.** `CLAUDE.md` + `docs/` are unusually good — the project is
  re-enterable after months away, which is the real longevity test.

### Structural weaknesses

1. **Babel-in-the-browser is the #1 startup cost.** ~136 KB of JSX is recompiled
   by `@babel/standalone` (a ~2–3 MB script itself) on **every page load**, on the
   main thread, before anything renders. This is seconds of white screen on a
   laptop, worse on mobile.
2. **Everything loads before first paint.** All 12 data files are blocking
   `<script>` tags in `<head>` — the browser parses 13 MB of JS (3.65 MB over the
   wire) before the shelf exists. The Reader-only overlays (OMDb plots, TMDB
   synopses, notes) are ~70% of that and aren't needed until a click.
3. **`omdb_data.js` ships dead weight.** 5.8 MB raw, but the app reads only
   `imdbRating/imdbVotes/Poster/imdbID/Ratings/Metascore/Awards/Plot/PlotShort`.
   Fields never read (`Rated`, `Released`, `Runtime`, `Genre`, `Director`,
   `Writer`, `Actors`, `Language`, `Country`, `DVD`, `BoxOffice`, `Production`,
   `Website`, `Response`, `Title`, `Year`, `Type`) plus the `Plot`/`PlotShort`
   double-ship are pure dead weight. Measured: a writer-side whitelist (keep the
   9 read fields, ship one plot string) yields **1.95 MB raw / 0.54 MB gz — a 66%
   cut with zero behaviour change**. And since `omdb_cache.json` is committed,
   this is rebuildable on THIS PC, no API key needed.
4. **One 2,600-line JSX file.** Still navigable thanks to comments, but it's at
   the edge. The no-build constraint blocks `import`, but multiple
   `<script type="text/babel">` files would work — or fall out naturally from
   fix #1 (precompile) which permits real modules.
5. **CDN dependency** (unpkg/jsdelivr for React, Babel, d3). An outage or
   version-yank takes the site down. Already on the backlog; still open.
6. **A React memo footgun:** `App()` builds `shelves` unmemoized (line ~2070), and
   `preChipShelves` lists it as a dependency — so the 4,600-item filter re-runs on
   every App render (every keystroke, modal open, mode switch). Harmless today,
   but it silently defeats the memo. One `useMemo` fixes it.
7. **No data-integrity gate.** The `id` join key is load-bearing for 9 overlays,
   but nothing checks for duplicate ids, orphaned overlay keys after a rename, or
   schema drift. `audit_enrich.py` measures coverage, not integrity.

### Verdict

The architecture is **fit for purpose and worth keeping**. Its ceiling (~10k items,
single author, static hosting) is far away. The weaknesses are all fixable within
the current model — none call for a rewrite, a framework, or a server.

## 3. Performance — measured

| File | Raw | Gzipped | Notes |
|---|---|---|---|
| `omdb_data.js` | 5.8 MB | 1.6 MB | 66% trimmable, measured (§2.3) |
| `cast_data.js` | 2.8 MB | 0.75 MB | cast capped at 22 — reasonable |
| `wishlist_cast.js` | 1.7 MB | 0.45 MB | |
| `tmdb_data.js` | 1.0 MB | 0.4 MB | Reader-only |
| `imports.js` | 0.9 MB | 0.19 MB | |
| everything else | ~0.8 MB | ~0.25 MB | |
| **Total data** | **13.0 MB** | **3.65 MB** | all blocking, in `<head>` |

Plus CDN: Babel standalone (~2–3 MB), React+DOM prod (~170 KB), d3 (~280 KB).

**Startup fix package** (ordered by value; all no-API, doable here):

1. **Precompile the JSX** in `build_all.py` (single `npx esbuild culture-v2.jsx
   --loader:.jsx=jsx --minify` call, output `culture-v2.js`), drop Babel standalone
   from `index.html`. Keeps the edit→refresh loop (build_all already runs per
   change). Likely the largest single win — removes both the 2–3 MB download and
   the per-load compile.
2. **Trim OMDb fields** at the writer (§2.3). −3.9 MB raw / −1.1 MB gz, measured.
3. **Defer the Reader-only overlays.** `omdb_data` / `tmdb_data` / `filmweb_notes`
   / `notes_en` aren't needed for first paint. Load them with `defer` + a tiny
   "re-enrich when ready" state bump, or inject after `DOMContentLoaded`. First
   shelf render then needs only ~1.2 MB gz.
4. **`defer` everything else** so HTML parsing isn't blocked (scripts with `defer`
   preserve execution order).
5. **Poster srcset:** grid covers at `w342`, Reader at `w500/w780` (string rewrite
   on `image.tmdb.org` URLs). Add `<link rel="preconnect" href="https://image.tmdb.org">`.
6. Self-host React/d3 (backlog item — folds into #1 since esbuild can bundle them).

Interaction performance is already good; the only code smell is §2.6.

## 4. Pipeline & repo health

- **Works well:** runbook discipline, `--ids=` targeting, resumability, committed
  OMDb cache as quota ledger.
- **Add `validate.py`** (fast, no network): duplicate `id`s across the three data
  files · overlay keys with no matching item (orphans after renames/deletes) ·
  items missing required fields · region codes absent from `REGION_NAMES`.
  Run it inside `build_all.py` before the cache bump; fail loudly.
- **Naming pass** (backlog: `fetch_*`/`build_*`/`edit_*`/`audit_*`) — still worth
  it, cheap, do it opportunistically.
- **Repo weight:** committed caches (`omdb_cache.json` 5.8 MB) and generated
  overlays mean history grows ~10 MB per full regen cycle. Tolerable for now;
  if clone times start to annoy, `git gc` + consider Git LFS for `*_cache.json`.
  Not urgent.
- **Two-PC workflow:** API keys live on the other PC. The split that works:
  **this PC = code, perf, docs, source-JSON curation, OMDb-cache-derived rebuilds**
  (the cache is committed, so `update_omdb.py`'s writer can be re-run here);
  **API PC = anything hitting TMDB/IGDB/OMDb-fetch/OpenLibrary**, i.e. new-item
  enrichment and the new fetchers proposed below. Committing here and pulling
  there is exactly right.

## 6. Feature roadmap — the "library on steroids"

Grouped by the driver, roughly ordered by value within each group. Items marked
🔑 need the API PC; unmarked ones run entirely here.

### 6a. Discovery (new material)

- **Outward Discovery tab** (already parked in backlog — promote it). Seed TMDB
  `/recommendations` from every item rated ≥9, aggregate + score by how many
  seeds recommend it, subtract seen+wishlist, write a review file → curate into
  `wishlist.js`. 🔑 for the fetch; the tab itself is local code. The same pattern
  extends to games via IGDB `similar_games` and anime via AniList.
- **Gap finder.** Cross-reference acclaimed-canon lists (IMDb Top 250, Sight &
  Sound, TSPDT — all fetchable or static) against the seen library, ranked by
  similarity to your taste clusters. "Blind spots you'd probably rate 9."
- **Wishlist prioritizer.** The Reader's similarity scorer already works; run it
  globally — score every wishlist item against your ≥9 seen items and surface a
  "you'll probably love these next" rail. Pure local compute, no API.
- **Franchise/series completion.** TMDB collections + `igdbFranchise`: "you've
  seen 4/6 of this collection" chips, and a wishlist auto-suggest for the missing
  entries. 🔑

### 6b. Analysis & analytics

- **Taste-profile overhaul** — the plan in `taste_profile_plan.md` is sound;
  build order as written (restructure → subtag chips → Palette explorer →
  signature tags). For **signature tags**, the baseline question answers itself
  without external data: over-representation vs the library's own global tag
  frequency (log-odds), which is exactly the "your fingerprint" framing.
- **People dossiers.** Click a director/actor → mini-page: their items in your
  library, your avg vs IMDb avg for them, timeline, **completion** ("7/13 of
  Kubrick's features seen" via TMDB person credits 🔑 or Wikidata).
- **Year in review.** Auto-generated per-year recap from `watchedDate`: counts,
  hours, top-rated, discoveries (gems), rating distribution vs previous years,
  busiest week. All local; the ActivityHeatmap data already exists.
- **Ratings-drift view.** Your rating vs date-rated scatter: are you getting
  harsher? Do rewatch-era ratings differ? Local.
- **Cover-colour family** — `color_sort_ideas.md` path A stands: precompute
  overlay 🔑 (needs poster downloads), then colour sort, colour-wheel stat,
  ambient Reader tint, palette-over-time strip all become small follow-ons.

### 6c. Cross-referencing

- **Adaptation links.** Wikidata "based on" → connect Books ↔ Movies/TV within
  the library ("you read the book AND saw the film — ratings side by side").
  With 114 books this will find real pairs. 🔑 (SPARQL, no key)
- **Collaborator webs.** Same DP/composer/writer across your favourites —
  data is already in `cast_data.js`; a "hidden connections" panel or chord
  diagram (d3 already loaded) is pure local work.
- **Structured awards** (backlog) via Wikidata: per-award chips that are real
  data, filterable (`award:oscar`). 🔑

### 6d. Sorting & browsing

- **New sorts, all local & cheap:** IMDb-rating sort · popularity sort
  (`imdbVotes`) · **contrarian sort** (|your rating − community|, both
  directions) · "unenriched first" (maintenance view). Each is one branch in
  `sortItems` + a menu entry.
- **Colour sort** — after the overlay lands (6b).
- **`on:` provider filter for the seen library** too (re-watch planning), once
  watch-provider fetch covers the library. 🔑
- **Duration-accurate games** via HowLongToBeat (6a's ecosystem). 🔑

### 6e. Platform / infra (the enablers)

1. Startup fix package (§3) — do first; every other feature renders faster.
2. `validate.py` integrity gate (§4).
3. Precompile unlocks **file splitting**: `culture-v2.jsx` → modules
   (Reader, Stats, Explorer, ShelfRow, data-merge) with no runtime cost.
4. Service-worker cache (stretch): instant repeat loads + offline shelf.

## 8. Insight expansions — the narrative layer (Fable proposal, 2026-07-07)

§6 covers discovery, analytics and cross-referencing well; what it doesn't have is the
layer that makes Rotation distinctive — **derived narrative** (the site reads your history
back to you) — and it leaves Culture's three most *unique* data assets underused: the
personal notes corpus, the dialogue-mood overlay, and the hand-built badge taxonomy.
Against the market (Letterboxd, Goodreads, Backloggd, Serializd): they all have shelves,
ratings and stats; none of them can narrate, because none of them have this data.

### 8a. Stories for Culture (cheap — watchedDate + imports already carry the history)
Port the *concepts* (not the code) of Rotation's Phase-3 sessions layer:
- **Chapters of your watching** — era auto-segmentation on the region/medium/genre mix
  over `watchedDate` ("the J-horror year", "the 2021 games pivot", "the shorts deluge").
- **Obsession curves** — director/franchise binges detected from date clusters
  (N items by the same director inside a window → flameout vs slow-burn, same
  trajectory classes as Rotation).
- **On this day / year-permalink recaps** — §6b's "year in review" generalized: a daily
  "five years ago you were deep in Kieślowski" hook, plus shareable per-year pages.

### 8b. The notes corpus as data — "your critical voice" (unique asset #1)
~983 Polish reader notes (260 with EN redrafts) are a decade-long diary of *reactions*
that no other platform's users even have. Mine it:
- **Praise/punish vocabulary** — what words appear in your 10s vs your 6s; your recurring
  obsessions (atmosphere? endings? pacing?) surfaced as a fingerprint of what you actually
  value, vs what the badges *say* you value (the divergence itself is an insight).
- **Notes × badges cross-check** — items whose note reads "floored" but carry no impact
  badge (and vice versa) → curation queue, plus honest measurement of badge drift.
- Feeds the taste-profile Portrait (taste_profile_plan §1) with real language, not stats.

### 8c. Mood & impact diet (unique assets #2 and #3)
- **Dialogue-mood diet** — `script_mood.js` (NRC valence/emotion per film) × ratings ×
  time: do you rate dark-dialogue films higher? Did your valence diet shift by era?
  The "reads dark ≠ is dark" caveat stays front and center (label as dialogue reads).
- **Impact timeline** — the badge taxonomy's impact family (floored / devastating /
  lingers) plotted over watchedDate: *what hit you hardest, year by year* — the emotional
  autobiography the badges were always building toward. Pairs with the open taxonomy
  decisions in backlog.md (do those first so the timeline uses the settled axes).

### 8d. Hub-level joins — Rotation × Culture (nobody on the market can do this)
Same person, same hub, two behavioral datasets:
- **The soundtrack bridge** — Rotation already sees Spider-Verse segue chains and
  Reznor→Santaolalla transitions; join scrobbles to the film/game library ("you watched
  it, then you played the soundtrack for three months" — or the reverse: the music led
  you to the film). Composer/artist overlap is resolvable via names + Wikidata.
- **Binge × listening temporal joins** — `watchedDate` × Rotation's day-series: what you
  listened to *while* bingeing The Wire; whether heavy watching weeks suppress or shift
  music listening. Pure local compute on data both apps already ship.
- Architecturally: a small shared JSON contract at the hub level (each app exports a
  minimal `{date, id, kind}` event stream), consumed by either app's insight pass —
  keeps the apps self-contained per HUB.md while allowing the join at build time.

### 8e. Thematic twins across mediums (stretch; analog of Rotation's lyrical twins)
Embed synopses + tags + notes → nearest-neighbour pairs *across* mediums ("Disco Elysium
↔ True Detective"), surfaced in the Reader as "rhymes with…". Needs an embedding pass
(one-off, local cache like everything else); best after the tag/badge explorer ships so
the pairs are clickable into the Palette.

### Suggested placement in §7's sequence
8a + 8b are Phase-2-grade (no keys, pure local) — slot after the quick feature wins.
8c rides with the Taste tab phase (8c's impact timeline wants the settled badge axes).
8d is its own mini-phase at the hub level; 8e is stretch after Phase 3.
